Konzepte

Die ePA ist verteilt: zwei produktive Provider-Backends speichern die Akten der Versicherten. Anfragen an eine Akte müssen über einen verschlüsselten Tunnel (VAU) laufen und mit einer gültigen Leistungserbringer-Identität (SMC-B) autorisiert werden. Die Middleware abstrahiert all das.

Proxy
Eine konfigurierte Identität dieser Middleware — SMC-B-Zertifikat (Arztpraxis, Krankenhaus, Apotheke) plus ePA-Umgebung (dev, ref, test, prod). Adressiert über /api/proxies/{name}.
Provider
Ein ePA-Backend. Es existieren zwei produktive Provider, identifiziert durch die Nummern 1 und 2. Welcher Provider die Akte eines Versicherten hostet, ist nicht öffentlich bekannt.
VAU
Vertrauenswürdige Ausführungsumgebung — verschlüsselter Tunnel zum Provider, in dem die Akten-Inhalte für den Provider unsichtbar bleiben. Die Middleware öffnet pro Proxy und Provider eine VAU-Session und hält sie offen.
Insurant
Ein Versicherter mit ePA, identifiziert durch eine KVNR (z.B. X110611629). Die Middleware ermittelt automatisch, bei welchem Provider die Akte liegt, und erzeugt die Befugnis für den Leistungserbringer.

Authentifizierung

Anfragen an diese Middleware werden über den HTTP-Header Authorization autorisiert. Wird die README-Seite mit einem solchen Header geladen, erscheint der Header automatisch in allen unten gezeigten curl-Befehlen.

Die Middleware selbst authentifiziert sich gegenüber gemIDP und ePA-Provider eigenständig mittels des im Proxy hinterlegten SMC-B Zertifikats. Für Clients ist dieser Schritt transparent.

Proxies

Jeder Proxy bündelt eine Leistungserbringer-Identität und eine ePA-Umgebung. Alle Anfragen starten mit dem Pfad-Präfix /api/proxies/{name}, der bestimmt, welche Identität verwendet wird.

Providers

Jeder Proxy unterhält zu beiden ePA-Providern eine VAU-Session, die im Hintergrund automatisch geöffnet, überwacht und erneuert wird. Diese Sessions werden für alle Insurant-Anfragen wiederverwendet, sodass keine Initialisierungs-Latenz pro Aufruf entsteht.

Direkte Provider-Anfragen umgehen die automatische Akten-Auflösung und richten sich an einen konkret benannten Provider {n} = 1 | 2:

  • /api/proxies/{name}/providers/{n}/… — unverschlüsselt zum Provider (für öffentliche Endpunkte wie Informationen oder Status)
  • /api/proxies/{name}/providers/{n}/vau/… — durch den VAU-Tunnel
GET/api/proxies/{name}/providers

Aktueller Stand der Provider-Sessions des Proxy — Base-URL und Zeitpunkt der Session-Öffnung.

Insurants

Der Insurants-Pfad ist der bequemste Weg, eine konkrete Akte anzusprechen: man gibt nur die KVNR an, die Middleware übernimmt die Routenermittlung, die Befugnis-Erteilung (Entitlement) und das VAU-Tunneling.

So läuft eine Anfrage intern ab:

  1. Erste Anfrage für eine KVNR: die Middleware fragt parallel beide Provider, welcher die Akte hält.
  2. Beim zuständigen Provider erzeugt sie eine Befugnis (Entitlement) für den im Proxy konfigurierten Leistungserbringer.
  3. Das Mapping KVNR → Provider wird gecached, sodass Folge-Anfragen direkt zum richtigen Backend gehen.
  4. Die Anfrage wird inklusive Method, Header, Query und Body durch den VAU-Tunnel weitergeleitet; die Antwort wird transparent zurückgegeben.

Pfad-Schema: /api/proxies/{name}/insurants/{kvnr}/vau/<Provider-Pfad>

Alles nach dem /vau/ ist der originale Provider-Pfad — die offizielle ePA-Dokumentation ist hier 1:1 anwendbar. HTTP-Method und Body werden ebenfalls durchgereicht.

GET/api/proxies/{name}/insurants/{kvnr}/vau/epa/medication/render/v1/eml/xhtml

Gerenderte Medikationsliste (eML) als application/xhtml+xml — geeignet zur direkten Anzeige im Browser.

Fehler

Fehler werden mit passenden HTTP-Statuscodes signalisiert. Direkte Provider-Antworten werden 1:1 durchgereicht (siehe das 404-Beispiel oben bei den Consent-Entscheidungen). Schlägt die automatische Akten-Auflösung über die /insurants/{kvnr}/vau/…-Route fehl, antwortet die Middleware mit einem zusammengesetzten Fehler beider Provider:

Beispiel — unbekannte KVNR 502 Bad Gateway · application/json
